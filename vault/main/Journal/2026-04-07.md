### GLM thinking leak — investigação e correção

**Causa raiz identificada:** `reasoning: false` no model spec é só metadata do OC — não suprime thinking na Z.ai API.

**Correções aplicadas (em ordem):**

1. `thinkingDefault: "off"` + `reasoningDefault: "off"` por agente (main, palmeiras, crypto-reports, crypto-posts) ✅
2. `agents.defaults.thinkingDefault: "off"` (global baseline) ✅
3. `thinking: "off"` em todos os `sessions_spawn` (21 calls em 5 arquivos de instructions + duplicados) ✅
4. `params.thinking: "off"` no model catalog para glm-5.1, glm-5, glm-4.7-flash, glm-4.5-flash ✅

**Incidente durante a correção:**
- Primeira tentativa usou `runtime: { thinkingDefault, reasoningDefault }` → gateway quebrou (`runtime` só aceita `"embedded"` | `"acp"` como string enum)
- Revertido imediatamente via backup, gateway restaurado em ~2 minutos
- Lição: sempre validar campos no `gateway.err.log` antes de confirmar

## Decisões Tomadas

- **Vault como padrão obrigatório** para toda criação/edição de instructions no OC — documentado em `AGENTS.md` e `_vault-rules.md`
- **`subagents.maxConcurrent: 2`** como limite conservador para Z.ai — pode aumentar se Z.ai liberar concorrência no plano
- **`params.thinking: "off"`** aplicado apenas nos modelos GLM com thinking habilitado por padrão (não nos modelos flash free que já estão off)
- **Vault do claude-bot** (`~/claude-bot/vault/`) confirmado como local de persistência de conhecimento navegável no Obsidian

## Informações Aprendidas

- O campo `runtime` em `agents.list` no OC é enum (`"embedded"` | `"acp"`), não objeto
- `thinkingDefault` e `reasoningDefault` ficam diretamente no agent object, não dentro de `runtime`
- `reasoningDefault` **não é válido** em `agents.defaults` — só por agente
- O plugin `zai` no OC lida com a tradução de `thinkingDefault: "off"` → `thinking: {type: "disabled"}` no payload da API
- Z.ai Coding Plan tem limite de 1 request simultâneo — `maxConcurrent: 8` causaria rate limits constantes nos pipelines
